Patient waitlist — three design variants

Soft-launch scenario: DNS is flipped to theravoca.com, therapist signup is fully live so we can recruit Idaho providers, but the patient intake is replaced with a waitlist signup until we hit our soft-launch supply target (~30-50 active therapists). Switch is admin- toggleable so we can flip patient intake live the moment we're ready, no redeploy needed. Three variants below trade-off how much we ask the patient up front; pick one and I'll implement.

Variant A — Minimal. Patient gives only email + ZIP. Lowest friction; fastest signup. Trade-off: we have nothing to sort by when matching opens, so the first wave of patients gets matched in arrival order regardless of fit. Best if you expect a small waitlist (<100 entries).

Implementation notes (read before picking)

How the waitlist toggles on/off

  • Admin setting: patient_intake_mode = "waitlist" | "open" (lives in app_config alongside the existing intake flags).
  • Frontend reads the mode from /api/config/patient-intake-mode on mount; routes to the waitlist screen or the full intake form accordingly. No redeploy required to flip.
  • Backend hard-rejects the existing POST /api/requests with a 503 when mode is waitlist, so a curl'd submission can't sneak past the UI gate.

Where waitlist signups land

  • New Mongo collection: patient_waitlist with fields email, zip, primary_issue, format_pref, first_name, created_at, source_ref.
  • Admin gets a new "Patient waitlist" panel under Testing with: count, recent signups, CSV export, and a "Convert N from waitlist → notify them" button that fires the go-live email when intake opens.
  • Per-IP + per-email rate limit (5/hour) to prevent spam signups. Cloudflare Turnstile gate same as the current intake.

Therapist signup pathway stays unchanged

  • /therapists/join remains fully functional regardless of patient mode. The landing's "For therapists" CTA is the same as today.
  • We add a small "Therapist? Join now and reach our growing patient waitlist" strip under the landing hero (shown in all 3 mockups) so therapists feel the supply-side pull while the patient side is in waitlist mode.

Day-of-launch flip

  • Admin clicks "Open patient intake" in the new panel. Mode flips to open.
  • Cron fires the "we're open" email to every waitlist signup. The email links them straight into /?ref=waitlist&email=<their email> with the intake form pre-filled from whatever we collected during waitlist signup (Variant B / C have more data to seed the form than A).
  • Once converted, the waitlist row is marked converted_at: <iso>; we keep the record for analytics (waitlist-to-conversion rate is a useful metric).
My recommendation: Variant B. Lowest cost to build (one screen, no multi-step state), gives admin enough signal to prioritize the first wave of matches by fit (not just arrival order), and the friction is barely higher than A (two extra dropdowns). Variant C feels great but is real engineering work for a soft-launch holding page. Variant A leaves you blind to who's on the list.

Scope estimate to ship the picked variant

  • A or B: ~half a day. One new React route, one new Mongo collection, one new admin panel card, one new cron, one new email template.
  • C: ~1-1.5 days. Same backend as B + the 3-step state machine + step navigation + the per-step animations to feel intentional.
